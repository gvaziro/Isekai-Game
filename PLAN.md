# Старт базы знаний The Last Summon с векторным поиском

## Summary

Цель первой итерации: запустить минимальную, но рабочую базу знаний лора, где канон хранится в Markdown-файлах с YAML-шапками, проверяется валидатором и индексируется локально в JSON/SQLite для структурного и семантического поиска.

Первый результат: 5-10 атомарных фактов по стартовой арке, 1 entity-файл для Хранителя 10-го этажа, locked-якоря канона, 1 референсная книга и простой скрипт индексации/поиска.

## Key Changes

- Создать структуру:
  - `world/facts/` — атомарные факты.
  - `world/entities/` — обзорные сущности.
  - `world/books/` — книги как отдельный канал лора.
  - `world/locked/` — неприкосновенные канонические якоря.
  - `system/schemas/` — схемы YAML-метаданных.
  - `changelog/` — журнал изменений лора.

- Зафиксировать минимальные схемы:
  - `fact`: `id`, `title`, `type`, `tier`, `tags`, `truth_level`, `related`, `locked`, `canonical_source`, `created_at`, `last_validated`.
  - `entity`: `id`, `title`, `entity_kind`, `canonical_facts`, `disputed_facts`, `open_questions`, `related_entities`, `locked_aspects`.
  - `book`: `id`, `title`, `category`, `truth_level`, `location`, `contains_facts`, `unlocks`.

- Перенести твёрдый канон из текущих документов в `locked/`:
  - деревня отрезана туманом;
  - туман связан с Хранителем 10-го этажа;
  - победа над Хранителем рассеивает туман;
  - после победы открывается путь наружу и более глубокая ветвь катакомб;
  - герой является исекай-перерождённым, но его прошлое выбирается игроком.

- Сделать первый референсный набор:
  - entity `the_keeper`;
  - facts про туман, катакомбы, 10-й этаж, печать, слухи жителей;
  - одну книгу/запись, например деревенскую хронику или запрещённую заметку о Хранителе.

- Реализовать локальный индекс:
  - Markdown остаётся источником правды.
  - Скрипт читает YAML + body, валидирует поля, собирает `knowledge-index.json` или SQLite.
  - Индекс хранит `id`, `type`, `title`, `tags`, `tier`, `truth_level`, `related`, `body`, `source_path`.
  - Векторный слой добавляется рядом: `embedding` или отдельная таблица/файл embeddings.
  - На первом этапе поиск должен поддерживать оба режима: по тегам/related/id и по похожести текста.

## Implementation Steps

1. Сначала финализировать схемы и папки, без генерации большого контента.
2. Затем вручную разложить стартовую арку на 5-10 атомарных фактов.
3. Добавить locked-файлы как канонические запреты для будущих агентов.
4. Собрать entity `the_keeper`, который ссылается на эти факты.
5. Добавить одну книгу, связанную с 1-2 фактами.
6. Написать валидатор:
   - проверяет обязательные поля;
   - проверяет уникальность `id`;
   - проверяет, что `related`, `contradicts`, `contains_facts`, `canonical_facts` указывают на существующие id;
   - блокирует изменение `locked: true` без явного режима override.
7. Написать индексатор:
   - пересобирает локальный JSON/SQLite индекс;
   - печатает статистику: число фактов, книг, entities, битые ссылки, locked-файлы.
8. Написать простой CLI-поиск:
   - `search "туман хранитель"` для семантики;
   - `search --tag туман`;
   - `search --related the_keeper`;
   - `search --tier village_common`.

## Test Plan

- Проверить, что валидатор падает на:
  - пропущенный `id`;
  - дублирующийся `id`;
  - битую ссылку в `related`;
  - неизвестный `truth_level`;
  - попытку изменить locked-факт без override.

- Проверить поиск:
  - запрос “почему деревня заперта” находит факты про туман и Хранителя;
  - запрос “10 этаж” находит стартовую арку и entity Хранителя;
  - структурный поиск по `related: the_keeper` возвращает все связанные факты;
  - книга возвращается отдельно от фактов, но показывает `contains_facts`.

- Проверить пригодность для NPC:
  - Елена видит только `village_common` и релевантные деревне факты;
  - скрытые/forbidden факты не попадают в обычный NPC-контекст;
  - locked-канон нельзя случайно переопределить новым фактом.

## Assumptions

- Первый горизонт: 1-2 недели, цель — рабочий фундамент, не массовое наполнение.
- Источник правды — Markdown + YAML, а не база данных.
- Первый индекс — локальный JSON/SQLite, без отдельного Qdrant/Chroma-сервера.
- Большой контент, генеративные горожане, автоматическое распространение слухов и полноценные AI-агенты откладываются до момента, когда 5-10 фактов стабильно валидируются и ищутся.
- Кодовая часть проекта сейчас недоступна из этой рабочей области, поэтому перед реализацией нужно будет сверить фактический стек проекта и команды запуска.
