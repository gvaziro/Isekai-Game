# Дизайн системы базы знаний и AI-NPC

Этот документ описывает архитектуру и принципы организации базы знаний игры, систему AI-NPC, работу с книгами как с параллельным каналом лора и инфраструктуру AI-агентов, поддерживающих канон.

Документ является рабочим. Он отражает текущее видение и будет обновляться по мере проработки системы.


## 1. Концепция

Игра строится вокруг двух ключевых идей:

**AI-NPC с распределёнными знаниями.** У каждого NPC есть LLM-агент и свой собственный, ограниченный срез информации о мире. Никто из NPC не знает всё, и ни один NPC не выдаёт игроку лор как «энциклопедия». Знания распределены между жителями деревни, странниками, книгами и средой.

**Большая база знаний с векторным поиском.** Лор хранится в виде отдельных файлов с метаданными. При разговоре NPC получает не «весь мир», а только релевантные ему чанки, отфильтрованные по его роли, кругу посвящения, личным фактам и текущему контексту разговора.

**Игрок открывает мир через диалоги и чтение.** Прогрессия знаний это отдельный пласт геймплея, наравне с боями и квестами. Часть истины невозможно узнать без книг. Часть невозможно узнать без доверительных отношений с конкретным NPC.

**Лор пишется и поддерживается ИИ-агентами.** Сам пользователь задаёт инварианты, направления и проверяет ключевые решения. Расширение, шлифовка и проверка консистентности делается агентами. Это требует особой архитектуры, потому что ИИ склонен к дрейфу канона и самосогласованным галлюцинациям.


## 2. Базовые принципы

**Атомарность.** Каждый факт это отдельная единица: его можно процитировать, изменить, связать с другими, удалить. Длинных «статей» в базе знаний быть не должно, кроме обзорных entity-файлов с особой ролью.

**Распределённое знание.** Один и тот же факт может быть известен нескольким NPC, но никогда всем. Полная картина мира никогда не доступна одному источнику.

**Множественные источники истины.** Один и тот же факт может быть подан через разные каналы (разные NPC, разные книги) с разной степенью точности. Это не противоречие, это фича.

**Уровни истинности.** Любой факт в базе имеет атрибут достоверности: это правда, искажение, слух, легенда, ложь. Игрок может знать факт, но не знать его truth_level, пока не сравнит с другими источниками.

**Прогрессивное раскрытие.** Знания открываются игроку постепенно: через доверие к NPC, через чтение книг, через события мира. NPC может знать что-то, но не говорить, пока не выполнены условия.

**Жёсткая ссылочная целостность.** Если факт ссылается на сущность или другой факт, эта ссылка должна быть валидна. Это критично, когда лор пишут агенты.


## 3. Архитектура хранения лора

### 3.1. Гибридная схема

База знаний состоит из двух типов файлов:

**Атомарные факты (`facts/*.md`).** Минимальная единица знания. Один файл = одно утверждение или одна тесно связанная группа утверждений на одну тему. Эти файлы являются чанками для векторного поиска во время игры. NPC получает их в контекст при разговоре.

**Обзорные сущности (`entities/*.md`).** Один файл на крупную сущность мира: персонаж, локация, организация, концепт. Содержит структурированный обзор: «что мы знаем», «открытые вопросы», «связанные факты». Эти файлы **не используются для NPC напрямую**, они нужны:

- автору как точка истины при ручной правке
- AI-агенту-писателю как контекст перед добавлением новых фактов
- AI-агенту-валидатору для проверки консистентности

Атомарные факты являются первичными. Обзорные сущности это производное представление, которое в идеале должно быть автоматически синхронизируемо с фактами агентом-смотрителем.

### 3.2. Почему именно гибрид

Чистая атомарность даёт идеальный векторный поиск, но не даёт автору и агентам цельной картины. При длительной генерации лора начинается дрейф: один и тот же персонаж в разных фактах постепенно описывается всё более несогласованно.

Чистые «статьи» удобны для чтения, но плохо чанкуются. Поиск выдаёт большие куски, NPC получает в контекст лишнее.

Гибрид решает обе проблемы: NPC и игроку отдаются атомы, агентам и автору отдаётся обзор.

### 3.3. Предлагаемая структура папок

```
world/
  facts/                 атомарные факты, основной материал для векторного поиска
  entities/              обзорные файлы по сущностям
  npcs/                  файлы конкретных именных NPC
  archetypes/            архетипы для генеративных горожан (рыбак, ребёнок, бабка, и т.д.)
  books/                 тексты книг как отдельные чанки
  locked/                заблокированные канонические якоря, неприкосновенные для агентов
system/
  prompts/               системные промпты для NPC-агентов и агентов-редакторов
  schemas/               YAML-схемы фактов, NPC, книг
  rules/                 правила для LLM (текущий WORLD_ARC_PROMPT.md и его расширения)
changelog/
  YYYY-MM-DD-*.md        запись каждого изменения, сделанного агентом
```

Папка `locked/` это критическая идея. Туда кладутся файлы, которые агент не имеет права изменять без явного разрешения автора: ключевые сюжетные точки, законы магии, инварианты стартовой арки. Поле `locked: true` дублируется в метаданных самого файла. Любая попытка изменения должна отклоняться валидатором.


## 4. Модель данных

### 4.1. Схема атомарного факта

YAML-шапка в начале каждого `facts/*.md`:

```yaml
---
id: fog_origin_legend
title: Легенда о происхождении тумана
type: fact

# распределение знания
tier: village_common              # village_common | village_rumor | professional | hidden | forbidden
tags: [туман, хранитель, легенда, деревня]
known_by_roles: [старейшина, жрец, бард]
known_by_npcs: [npc_baba_marta]   # персональные носители сверх ролевого пула
known_by_books: [book_village_chronicle]

# истинность
truth_level: partial               # true | partial | distorted | false | encoded
contradicts: [fog_origin_truth]    # факты, с которыми этот конфликтует по содержанию
related: [the_keeper, catacombs_floor_10]

# условия раскрытия
requires_trust: 0                  # минимальный уровень доверия NPC для произнесения
unlocks_after: null                # id события или факта, после которого знание становится доступно
trigger_topics: [туман, выход, легенды]   # темы, на которые NPC заговорит сам

# канонический статус
locked: false                      # неизменяемый якорь канона
canonical_source: WORLD_ARC_PROMPT.md
created_by: human
created_at: 2026-05-06
last_validated: 2026-05-06
---

Деревенские старики рассказывают, что туман на дороге появился, когда из-под земли...
```

Не все поля обязательны для каждого факта, но базовый набор (`id`, `title`, `tier`, `tags`, `truth_level`, `related`, `locked`) должен быть всегда.

### 4.2. Схема обзорного entity-файла

```yaml
---
id: the_keeper
title: Хранитель 10-го этажа
type: entity
entity_kind: character             # character | location | organization | concept | item

# что мы про это знаем
canonical_facts: [keeper_origin, keeper_role_jailer, keeper_appearance]
disputed_facts: [keeper_motivation, keeper_age]
open_questions:
  - Знал ли хранитель о своей роли тюремщика
  - Был ли он человеком до того, как стал хранителем
  - Что именно он удерживает под собой

# связи
related_entities: [catacombs, the_seal, deeper_floors]
locked_aspects: [роль_тюремщика, рассеивание_тумана_после_смерти]
---

Краткий обзор сущности для автора и агентов. Не используется напрямую в диалогах NPC.
```

Поле `locked_aspects` это перечень тех аспектов сущности, которые **нельзя менять** при генерации новых фактов. Агент, добавляющий факт о хранителе, обязан проверить, не противоречит ли он этим аспектам.

### 4.3. Схема NPC

```yaml
---
id: npc_blacksmith_orin
title: Орин, кузнец
type: npc
npc_class: named                   # generic | merchant | named | wanderer

# профиль
role: кузнец
archetype: молчаливый_мастер
location: деревня_центр
tier_access: [village_common, village_rumor, professional]

# знания
known_facts: [forging_basics, iron_sources_local, dwarf_tools_legend]
personal_secrets: [orin_ex_dungeon_delver, orin_lost_brother]
disagrees_with:
  - [npc_priest, fog_origin_legend]   # Орин не верит официальной версии

# поведение
truth_reliability: high            # high | medium | low
talkativeness: low                 # сколько фактов выдаёт за разговор
trust_gates:
  50: [orin_ex_dungeon_delver]     # на доверии 50 раскрывает свой секрет
  80: [orin_lost_brother]

# квесты
quest_giver_for: [quest_lost_brother]
---

Биографическая справка: где родился, как попал в деревню, что любит, что ненавидит. Используется как часть системного промпта при разговоре с этим NPC.
```


## 5. Система NPC

### 5.1. Типы NPC

**Generic (обычные горожане).** Не имеют индивидуальных файлов. У каждого есть `archetype` (рыбак, фермер, ребёнок, торговка зеленью, пьяница, бабка-сплетница). При разговоре NPC получает 1-3 факта из общего пула `tier: village_common` или `tier: village_rumor`, отфильтрованных по тегам, плюс flavor-инструкции архетипа. Это даёт ощущение живой деревни без необходимости прописывать каждого жителя руками.

**Merchant (торговцы).** Имеют свой файл, базовую историю, профессию, инвентарь, но обычно ограниченный круг знаний по своей сфере. Могут иметь 1-2 личных секрета.

**Named (именные).** Полноценные персонажи с проработанной биографией, личными секретами, расширенным кругом знаний, квестами, отношением к герою. Это основной носитель сюжетных знаний в деревне.

**Wanderer (странники).** Эпизодические NPC, появляющиеся в деревне на короткое время. Носят знания о внешнем мире, недоступные жителям. До победы над туманом их крайне мало (объясняет изоляцию). После победы поток странников растёт. Хороший канал для подсветки будущих глав игры без нарушения правила про действующий туман.

### 5.2. Модель знания

Любой именной NPC получает доступ к фактам по нескольким каналам:

**Ролевой пул.** Кузнец автоматически получает доступ к фактам с тегами «ремесло», «металл», «оружие», и так далее, в пределах своих `tier_access`.

**Tier access.** NPC ограничен по уровню посвящения. Обычный житель никогда не получит доступ к `tier: forbidden`, как бы хорошо к нему ни относился игрок.

**Личные факты (`known_facts`).** Сверх ролевого пула NPC может знать 2-5 уникальных фактов, которые никто кроме него не знает. Это делает каждого NPC ценным: у него есть что-то своё.

**Личные секреты (`personal_secrets`).** Подмножество личных фактов, которые NPC **активно скрывает**. Они раскрываются только при достижении нужного `trust_gates` или через квестовые триггеры. Это материал для арок и реигрного контента.

### 5.3. Доверие и активные/пассивные знания

NPC может знать факт, но **не говорить его**, пока не выполнены условия. Это критично, чтобы первая встреча не превращалась в info dump.

**Trust gates.** Чем выше доверие героя у NPC, тем глубже tier открывается. Доверие растёт через диалоги, услуги, подарки, общие испытания.

**Активные знания.** Факт с `trigger_topics: [туман, легенды]` означает, что NPC может сам начать про это говорить, если разговор зашёл в эту область. Без триггера он молчит.

**Пассивные знания.** Факт без триггеров. NPC выдаст его только если игрок прямо спросит на эту тему, и только если есть доступ.

**События как триггер.** Поле `unlocks_after` позволяет привязать раскрытие факта к событию мира. Например, секрет о подземельном прошлом кузнеца открывается только после того, как игрок впервые спустился в катакомбы.

### 5.4. Сплетники против экспертов

Поле `truth_reliability` у NPC задаёт, насколько искажённо он передаёт известные ему факты.

**High.** Эксперт. Передаёт факт точно, как написан в базе. Знает мало, но достоверно.

**Medium.** Обычный житель. Может перепутать детали, добавить интерпретацию.

**Low.** Сплетница. Передаёт факт с искажением. На уровне промпта NPC получает инструкцию преувеличить, додумать или связать с не относящимися деталями.

Это создаёт игровой цикл: «иди к бабке, она расскажет много, потом к жрецу, он подтвердит или опровергнет». Один и тот же факт, услышанный из двух уст, ощущается совершенно по-разному.

### 5.5. Конфликты между NPC

Поле `disagrees_with: [[npc_id, fact_id]]` позволяет двум NPC иметь явно разные версии одного факта. Это автоматически создаёт интригу: «почему староста говорит одно, а жрец другое?». Игрок чувствует мир неоднозначным, и это **сильнее лорится в память**, чем согласованная подача.

### 5.6. Генеративные горожане в деталях

Архетип это файл вида `archetypes/old_gossip.md`:

```yaml
---
id: archetype_old_gossip
type: archetype
flavor: |
  Бабка лет под семьдесят. Знает всех, всё помнит, всё пересказывает с искажениями.
  Любит начинать с «я-то старая, мне можно сказать». Часто противоречит самой себе в одном предложении.
truth_reliability: low
talkativeness: high
preferred_tags: [слухи, семьи, старые_времена, дети]
tier_access: [village_common, village_rumor]
fact_count_per_dialog: [2, 4]
---
```

Конкретный NPC-горожанин на сцене получает случайное имя, случайный архетип, случайные факты из ролевого пула. Если игрок встретит его второй раз, система должна помнить, что это тот же NPC, и сохранять консистентность (тут понадобится отдельный механизм, см. открытые вопросы ниже).


## 6. Книги как параллельный канал

Книги это **отдельный, независимый от NPC канал** доставки лора. Часть истины мира в принципе невозможно узнать через разговоры. Это намеренный дизайн: самопознание героя как геймплейная задача, решаемая через исследование и чтение.

### 6.1. Категории книг

**Личные дневники жителей.** Субъективный взгляд. Факты иногда искажены эмоциями, но дают доступ к скрытым событиям, которые живые NPC не вспоминают.

**Официальные хроники и документы.** Сухой стиль, точные даты и имена, но интерпретация прилизана или вычеркнута.

**Запрещённые и утерянные книги.** Носители настоящей истины. Прячутся в катакомбах, тайниках, у отшельников. Высокий truth_level, но опасный доступ.

**Детские сказки и фольклор.** Зашифрованные древние факты под видом безобидных историй.

### 6.2. Книги в базе знаний

Каждая книга это отдельный файл в `books/`, с метаданными:

```yaml
---
id: book_diary_of_old_hunter
title: Дневник старого охотника
type: book
category: personal_diary           # personal_diary | official_chronicle | forbidden | folklore
truth_level: subjective            # subjective | official_distorted | hidden_truth | encoded_truth
location: дом_охотника_чердак
discovery_difficulty: easy         # easy | medium | hard | gated
unlocks: [fact_hunter_saw_keeper, quest_old_path]
contains_facts: [fact_hunter_saw_keeper, fact_forest_strange_signs]
---

Полный текст книги, разбитый на главы или записи. Каждая запись это потенциальный чанк для векторного поиска.
```

### 6.3. Правила работы с книгами

LLM **не должна выдумывать содержание книг на лету**. Если игрок спрашивает у NPC про книгу, которой нет в базе, NPC отвечает «не знаю» или «не слышал».

Книги, которые игрок прочитал, добавляются в его «прочитанное», и факты из них становятся доступны для его реплик. NPC должен учитывать это при ответах: «А, ты читал хроники старосты, тогда понимаешь...».

Запрещённые книги не лежат в стартовой деревне в открытом доступе. Если такая книга появляется в руках героя рано, это сюжетное событие, а не находка.

`unlocks` у книги может запускать новые квесты, изменения в диалогах NPC, новые опции в выборе.


## 7. AI-агенты, поддерживающие лор

Главный риск AI-генерации лора это **дрейф канона**. Агент случайно создаёт факт, противоречащий старому, забывает прописанные мелочи, или зацикливается на одних и тех же мотивах. Чтобы этого избежать, нужна жёсткая инфраструктура.

### 7.1. Разделение ролей

Один агент не должен и писать, и проверять. Это базовая проблема LLM-самооценки: агент в 80% случаев одобряет свою же работу.

Минимум три отдельных агента, с разными системными промптами и разными вызовами:

**Писатель.** Получает запрос на расширение лора (от автора или от смотрителя), выдаёт черновики новых фактов или правок к существующим. Не имеет права коммитить сам.

**Валидатор.** Получает черновик от писателя. Обязан перед одобрением:

- сделать векторный поиск похожих фактов и проверить отсутствие конфликтов
- проверить, что черновик не задевает `locked: true` факты и `locked_aspects` в entity-файлах
- проверить ссылочную целостность всех `related`, `contradicts`, `known_by_*` ссылок
- если есть конфликт, либо отклонить, либо потребовать явное добавление в `contradicts`

**Смотритель.** Регулярно (по расписанию или по триггеру) делает обходы базы и ищет проблемы. Не правит сам, формирует тикеты для писателя.

### 7.2. Lore locks как якоря канона

Канонические инварианты помечаются `locked: true` и складываются в папку `locked/`. Это:

- стартовая арка (туман, хранитель, 10-й этаж, рассеивание)
- природа исекаев и их роль в снятии печатей
- категории книг и truth_level каждой
- ключевые сюжетные повороты, заранее зафиксированные автором

Любая попытка агента изменить файл с `locked: true` или нарушить `locked_aspects` сущности должна автоматически блокироваться валидатором, без обсуждения.

### 7.3. Changelog и откаты

Каждое изменение в базе знаний (новый факт, правка факта, удаление, ребалансировка тегов) логируется в отдельный файл `changelog/YYYY-MM-DD-<id>.md`:

```yaml
---
date: 2026-05-06
agent: writer-v1
action: create_fact
target: fact_keeper_was_human_once
diff: |
  + создан новый факт о том, что хранитель когда-то был человеком
related_changes:
  - update_entity:the_keeper:open_questions
reason: запрос автора на проработку backstory хранителя
---

Полный текст изменения для возможности отката.
```

Без этого через месяц невозможно понять, откуда взялся какой-то факт и как откатить плохое решение агента.

### 7.4. Регулярные pass-ы смотрителя

Агент-смотритель запускает по расписанию проверки разного типа:

**Поиск противоречий.** Сравнить пары фактов с пересекающимися тегами и темами, найти семантические конфликты, не отмеченные в `contradicts`.

**Поиск orphan-сущностей.** Сущности, упомянутые в трёх и более фактах, но не имеющие своего entity-файла. Создать тикет.

**Поиск over/under-density.** Темы, по которым слишком много или слишком мало фактов относительно их сюжетной важности. Слишком плотные темы могут быть признаком зацикливания агента, разреженные - дырами в лоре.

**Проверка ссылочной целостности.** Все `related`, `contradicts`, `known_by_npcs`, `unlocks` должны указывать на существующие id. Битые ссылки это тикет.

**Проверка распределения знаний.** Если факт `tier: village_common`, у него должно быть несколько `known_by_roles`. Если `tier: hidden`, скорее всего один-два посвящённых. Если эти инварианты нарушены, тикет.

**Проверка `last_validated`.** Факты, не валидировавшиеся давно, прогоняются заново. Канон мог измениться.

### 7.5. Двойной индекс

Помимо векторного поиска по содержанию, держится **структурированный индекс** по тегам, ролям, сущностям, id.

Векторный поиск отлично работает для семантически близких запросов, но плохо для перечислений типа «дай мне все факты про хранителя». Структурный индекс по тегу `хранитель` или по `related: the_keeper` гарантирует полное покрытие.

NPC во время игры использует **оба**: векторный поиск по теме разговора плюс жёсткий фильтр по доступным тегам и tier.


## 8. Жизненный цикл факта

От идеи до использования в игре:

1. **Запрос на расширение.** Автор или агент-смотритель формулирует, что нужно добавить.
2. **Писатель готовит черновик.** Сначала делает векторный и структурный поиск по теме, читает связанные entity-файлы, читает locked-якоря. Только после этого пишет.
3. **Валидатор проверяет.** Если есть конфликты или нарушения lock, отправляет на доработку.
4. **Коммит и логирование в changelog.** Факт получает `created_at`, `created_by`, `last_validated`.
5. **Обновление обзорных entity-файлов.** Если факт затрагивает entity, его список `canonical_facts` обновляется. Это делает смотритель отдельным проходом.
6. **Распространение по NPC.** Если факт имеет `known_by_roles`, NPC с этой ролью получают к нему доступ автоматически. Если `known_by_npcs`, нужно явно добавить факт в файл NPC.
7. **Ввод в игру.** Игрок может встретить факт через диалог, чтение, событие.
8. **Периодическая ревалидация.** Смотритель время от времени перепроверяет факт на актуальность.


## 9. Открытые вопросы

Эти вопросы важны, но в этом документе ещё не решены. Их нужно проработать отдельно.

**Память сессии для генеративных горожан.** Если игрок встретил безымянного рыбака вчера и сегодня, должен ли это быть тот же рыбак с тем же именем и фактами, или каждый раз новый? Какой механизм запоминания.

**Эволюция знаний по ходу игры.** Если в мире произошло событие (хранитель убит, туман рассеян), как обновляются факты у NPC? Часть фактов получает новый tier, часть устаревает, появляются новые. Это требует системы «версий мира».

**Автоматическое распространение слухов.** Если что-то произошло, новость должна постепенно расползтись по жителям. Это означает, что `tier` у факта может меняться со временем, и не все NPC сразу получают доступ.

**Балансировка количества фактов на NPC.** Сколько фактов на одного NPC это «достаточно живо» и при этом «не перегружено»? Нужно эмпирически подобрать.

**Длина и структура промпта NPC.** При разговоре в контекст идут: системный промпт NPC + биография + N релевантных фактов + история диалога. Где оптимум по N, и как фильтровать.

**Защита от prompt-инъекций через игроков.** Игрок может попытаться через диалог заставить NPC выдать всё что знает, или притвориться другим NPC. Нужна защита на уровне системного промпта.

**Многоязычность.** Если игра планируется на нескольких языках, как организовать хранение лора. Один файл со всеми переводами, отдельные файлы по языкам, или генерация на лету.

**Версии и ветки лора.** При экспериментах с агентами иногда хочется попробовать альтернативную ветку канона без риска для основной. Имеет смысл подумать о git-подобной системе или маркерах ветвей.


## 10. Приоритеты на ближайшее время

Предлагаемый порядок проработки, чтобы не делать всё сразу:

1. Финализировать YAML-схемы факта, NPC и книги. Без этого писать содержательный лор бессмысленно: придётся переписывать.
2. Создать папку `locked/` и положить туда инварианты стартовой арки и расширений (исекаи, книги).
3. Прописать первую полноценную сущность как референс: например, хранителя 10-го этажа, со всеми связанными фактами и locked_aspects.
4. Прописать первого именного NPC по полной схеме (история + tier + личные секреты + знания + квест).
5. Прописать одну запрещённую книгу как референс категории.
6. Настроить системные промпты для трёх агентов (писатель, валидатор, смотритель) и протестировать пайплайн на маленьком объёме.
7. После того как пайплайн работает на 5-10 фактах, начать массовое расширение.

Документ будет дополняться по мере проработки.
